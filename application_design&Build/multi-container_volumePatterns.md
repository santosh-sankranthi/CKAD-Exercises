# CKAD Pattern Playbook: Volumes & Multi-Container Pod Design

This guide maps out all 12 core patterns for Persistent/Ephemeral Volumes and Multi-Container Pod Design. Recognizing the pattern instantly is the key to passing the CKAD.

---

## Tier 1: Highly Probable (The "Almost Guaranteed" Patterns)

### Pattern 1: The Sidecar & `emptyDir` Synergy
**[OVERLAP ALERT: Multi-Container Design + Ephemeral Volumes]**

* **Concept:** Two containers running in the same Pod sharing transient data. One generates/writes data, the other reads/ships it.
* **The Trick:** The exam will often require you to mount the exact same `emptyDir` volume to *different* directory paths (`mountPath`) inside each container.
* **The Realization:** When a prompt says "Container A creates a log file" and "Container B needs access to read that log file," your brain should immediately jump to `emptyDir`. Do not waste time provisioning PersistentVolumes.



> **CKAD Scenario:**
> Create a Pod named `log-processor`. 
> - Container 1 (`app`): Runs `sh -c 'while true; do date >> /var/log/app/data.txt; sleep 5; done'`.
> - Container 2 (`sidecar`): Runs `sh -c 'tail -f /var/log/read/data.txt'`.
> - Use a shared volume so the `sidecar` container reads the exact data generated by `app`.

<details>
<summary><b>View Solution (Workflow & YAML)</b></summary>

**Imperative Generation:**
`kubectl run log-processor --image=busybox --dry-run=client -o yaml -- sh -c 'while true; do date >> /var/log/app/data.txt; sleep 5; done' > pod.yaml`

**Declarative Edit (`pod.yaml`):**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: log-processor
spec:
  volumes:
  - name: shared-logs
    emptyDir: {}
  containers:
  - name: app
    image: busybox
    command: ["sh", "-c", "while true; do date >> /var/log/app/data.txt; sleep 5; done"]
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log/app
  - name: sidecar
    image: busybox
    command: ["sh", "-c", "tail -f /var/log/read/data.txt"]
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log/read
```
</details>
**Thing where you can go wrong**
> - labels names, mount names, mount paths.

### Pattern 2: The `InitContainer` Blocker
**[Concept: Multi-Container Pod Lifecycle]**

* **Concept:** A setup script or dependency check that must complete successfully *before* the main application is allowed to boot.
* **The Trick:** Placing `initContainers:` must be at the same indentation level as `containers:` under `spec:`.
* **The Realization:** Look for trigger words like "wait," "delay," or "ensure X is ready before Y starts." This mandates an InitContainer.



> **CKAD Scenario:**
> Deploy a Pod named `web-app`. The application must not start until a file named `/work-dir/ready.txt` exists. Add an init container that creates this file. Mount an `emptyDir` volume to share `/work-dir`.

<details>
<summary><b>View Solution (Workflow & YAML)</b></summary>

**Imperative Generation:**
`kubectl run web-app --image=nginx --dry-run=client -o yaml > init-pod.yaml`

**Declarative Edit (`init-pod.yaml`):**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  volumes:
  - name: workdir
    emptyDir: {}
  initContainers:
  - name: wait-for-it
    image: busybox
    command: ["sh", "-c", "touch /work-dir/ready.txt"]
    volumeMounts:
    - name: workdir
      mountPath: /work-dir
  containers:
  - name: web-app
    image: nginx
    volumeMounts:
    - name: workdir
      mountPath: /work-dir
```
</details>

### Pattern 3: Configuration Injection (ConfigMap/Secret)
**[Concept: Projected Ephemeral Volumes]**

* **Concept:** Mounting configuration data as physical files inside a Pod's filesystem.
* **The Trick:** Mounting a *specific key* from the ConfigMap as a specific filename without overwriting the directory's other contents. You must use `subPath`.
* **The Realization:** "As an environment variable" = `envFrom`. "As a file/volume" = `volumeMounts`.

> **CKAD Scenario:**
> Create a ConfigMap `app-config` (key `settings.conf`, value `color=blue`). Deploy a Pod `color-app`. Mount *only* the `settings.conf` file into `/etc/config/`.

<details>
<summary><b>View Solution (Workflow & YAML)</b></summary>

**Imperative Generation:**
1. `kubectl create configmap app-config --from-literal=settings.conf=color=blue`
2. `kubectl run color-app --image=nginx --dry-run=client -o yaml > config-pod.yaml`

**Declarative Edit (`config-pod.yaml`):**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: color-app
spec:
  volumes:
  - name: config-volume
    configMap:
      name: app-config
  containers:
  - name: color-app
    image: nginx
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config/settings.conf
      subPath: settings.conf # Mounts as a single file
```
</details>

## Tier 2: Medium Probability (Build, Format, & Fix)

### Pattern 4: The Full State Chain (PV -> PVC -> Pod)
**[Concept: Persistent Volumes]**

* **Concept:** Wiring up stateful storage that survives Pod deletion.
* **The Trick:** The PVC stays `Pending` if `accessModes`, `storageClassName`, or capacity do not perfectly match the PV. 
* **The Realization:** Requires data to "persist across Pod restarts."



> **CKAD Scenario:**
> 1. Create a PV `db-pv`: `1Gi`, `ReadWriteOnce`, `manual` storage class, hostPath `/mnt/data`.
> 2. Create a PVC `db-pvc`: request `500Mi` from `manual` class.
> 3. Create Pod `db-pod` mounting the PVC to `/data`.

<details>
<summary><b>View Solution (YAML)</b></summary>

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: db-pv
spec:
  capacity:
    storage: 1Gi
  accessModes:
    - ReadWriteOnce
  storageClassName: manual
  hostPath:
    path: /mnt/data
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: db-pvc
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: manual
  resources:
    requests:
      storage: 500Mi
---
apiVersion: v1
kind: Pod
metadata:
  name: db-pod
spec:
  volumes:
  - name: db-data
    persistentVolumeClaim:
      claimName: db-pvc
  containers:
  - name: redis
    image: redis
    volumeMounts:
    - name: db-data
      mountPath: /data
```
</details>

### Pattern 5: The Data Adapter
**[Concept: Multi-Container Design]**

* **Concept:** Transforming unstructured output using a secondary container.
* **The Trick:** Passing complex shell commands inside the YAML `args` array.
* **The Realization:** Identical to Pattern 1, but with a transformation command.

> **CKAD Scenario:**
> Add an adapter container that runs a `grep` command to filter for "ERROR" from a shared log file and outputs it to `stdout`.

<details>
<summary><b>View Solution (YAML snippet)</b></summary>

```yaml
  - name: adapter
    image: busybox
    command: ["/bin/sh", "-c"]
    args: ["tail -f /var/log/app.log | grep ERROR"] 
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log
```
</details>

### Pattern 6: Port Conflict Crash (Troubleshooting)
**[Concept: Multi-Container Networking]**

* **Concept:** Fixing `CrashLoopBackOff` in multi-container Pods.
* **The Trick:** Containers share `localhost` and cannot bind the same port.
* **The Realization:** Logs show `address already in use` or `bind failed`.

> **CKAD Scenario:**
> Pod with `nginx` and `httpd` is failing to start. Fix it.

<details>
<summary><b>View Solution (Workflow)</b></summary>

1. `kubectl logs broken-pod -c httpd` -> Output shows port 80 bind error.
2. `kubectl edit pod broken-pod` -> Change one container's port to `8080`.
</details>

---

### Pattern 7: Storage Troubleshooting
**[Concept: Volumes]**

* **Concept:** Fixing Pods stuck in `Pending` or `ContainerCreating`.
* **The Trick:** Identifying the broken link in the PV -> PVC -> Pod chain.
* **The Realization:** Always use `kubectl describe pod` to read the volume mount events.

> **CKAD Scenario:**
> A Pod is stuck in `Pending`. Fix the issue so it reaches `Running`.

<details>
<summary><b>View Solution (Workflow)</b></summary>

1. `kubectl describe pod stuck-pod`
2. Check events at the bottom. Usually, it says "persistentvolumeclaim 'xyz' not found".
3. Check the PVC name in the pod YAML vs what actually exists (`kubectl get pvc`) and fix the typo.
</details>

## Tier 3: Least Probable (The Edge Cases)

### Pattern 8: Volume Permissions (`securityContext` & `fsGroup`)
**[OVERLAP ALERT: Volumes + Pod Security]**

* **Concept:** Resolving write permission errors when a non-root container mounts a PV.
* **The Trick:** `runAsUser` solves process permissions; `fsGroup` solves filesystem ownership.
* **The Realization:** App logs say `Permission Denied` on a successfully mounted PVC.

> **CKAD Scenario:**
> Update a Pod so the mounted volume is owned by group ID `2000`.

<details>
<summary><b>View Solution (YAML snippet)</b></summary>

```yaml
spec:
  securityContext:
    runAsUser: 1000
    fsGroup: 2000 # Changes volume ownership
```
</details>

### Pattern 9: The Ambassador Proxy
**[Concept: Multi-Container Design]**

* **Concept:** Local proxy container routing external traffic for the main app.
* **The Trick:** Main app talks only to `localhost`, Ambassador talks to the world.



<details>
<summary><b>View Details</b></summary>

Rarely tested practically due to the time it takes to write proxy config files (like HAProxy) from memory. Understand the concept for multiple-choice scenarios.
</details>

---

### Pattern 10: Dynamic Provisioning (`StorageClass`)
**[Concept: Persistent Volumes]**

* **Concept:** Auto-provisioning cloud disks via PVCs.
* **The Trick:** You don't create the PV manually; the cluster does it.

<details>
<summary><b>View Details</b></summary>

You will likely just reference an existing `StorageClass` name in your PVC YAML under `storageClassName: <name>`. You won't write the actual `StorageClass` object from scratch on the CKAD.
</details>

---

### Pattern 11: Volume `subPath`
**[Concept: Volumes]**

* **Concept:** Mounting a single file into an existing container directory without hiding the directory's other files.

<details>
<summary><b>View Solution (YAML snippet)</b></summary>

```yaml
    volumeMounts:
    - name: config-volume
      mountPath: /etc/nginx/nginx.conf # Full path including filename
      subPath: nginx.conf # References the key in the volume
```
</details>

### Pattern 12: Ephemeral Debugging Containers
**[Concept: Multi-Container/Debugging]**

* **Concept:** Attaching a temporary container to a distroless Pod to inspect it.

<details>
<summary><b>View Solution (Imperative Command)</b></summary>

`kubectl debug my-pod -it --image=busybox --target=main-app-container`
</details>
